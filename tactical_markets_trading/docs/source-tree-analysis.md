## File-by-file (production path)

### `run_trading.py` — Entry orchestration

The **Entry task** entrypoint, scheduled at 08:35 AM CDT. Flow:

1. Load `.env` ([run_trading.py:66](../run_trading.py#L66)).
2. `today_signal()` scans [`tactical_markets/data/theses.jsonl`](../../tactical_markets/data/theses.jsonl), returns the latest `signal: true` record whose `as_of` is today UTC ([run_trading.py:17-25](../run_trading.py#L17-L25)). No signal → exit clean.
3. `already_traded(symbol)` queries Alpaca for any open position OR open order in `symbol` ([run_trading.py:28-38](../run_trading.py#L28-L38)). **Alpaca is the authoritative source** — local file lags if logging fails.
4. `submit_order(thesis)` ([src/order_builder.py:26-43](../src/order_builder.py#L26-L43)) submits a `MarketOrderRequest(symbol=buy_leg, notional=$10k, side=BUY, time_in_force=DAY)`.
5. `log_entry(order_result)` ([src/trade_logger.py:56-75](../src/trade_logger.py#L56-L75)) waits for fill, writes the entry row to `data/trades.jsonl` with NYSE-aware `exit_time_planned = +5 trading days`.
6. Pushover notification on success ([run_trading.py:59-62](../run_trading.py#L59-L62)). Any uncaught exception goes through Pushover via the `try/except` at the entrypoint ([run_trading.py:68-71](../run_trading.py#L68-L71)).

### `src/exit_manager.py` — Exit orchestration

The **Exit task** entrypoint, scheduled at 08:40 AM CDT. Flow:

1. Load `.env` ([src/exit_manager.py:114](../src/exit_manager.py#L114)).
2. `run_exits()` loads all records from `data/trades.jsonl` ([src/exit_manager.py:78-110](../src/exit_manager.py#L78-L110)).
3. For each record where `status == "open"` and `now >= exit_time_planned`:
   - `exit_position(record)` submits a market SELL for `fill_qty` shares ([src/exit_manager.py:43-75](../src/exit_manager.py#L43-L75)).
   - Waits for fill via [src/trade_logger.py:35-53](../src/trade_logger.py#L35-L53).
   - Computes `pnl_dollars`, `pnl_pct`.
   - **After the SELL has filled**, attempts benchmark capture via `yfinance` for SPY and the sell-leg (the loser-leg from the original thesis). Failures here leave `spy_return_pct` / `sell_leg_return_pct` as `null` but the close **always persists** ([src/exit_manager.py:68-74](../src/exit_manager.py#L68-L74)).
   - `save_trades(records)` rewrites the file with the closed record in place.
   - Pushover notification with P&L + benchmark comparison.
4. Per-trade exit failures are caught + reported but do not stop the loop ([src/exit_manager.py:102-107](../src/exit_manager.py#L102-L107)). Top-level crashes go through Pushover via the entrypoint try/except ([src/exit_manager.py:115-119](../src/exit_manager.py#L115-L119)).

### `src/alpaca_connector.py` — Auth + client factory

Two functions, ~30 LOC.

- `load_env()` ([src/alpaca_connector.py:8-17](../src/alpaca_connector.py#L8-L17)) — loads `.env` from the project root, validates `ALPACA_API_KEY` + `ALPACA_SECRET_KEY` are set, raises `RuntimeError` if not.
- `trading_client()` ([src/alpaca_connector.py:20-22](../src/alpaca_connector.py#L20-L22)) — returns `TradingClient(api_key, secret_key, paper=True)`. **The `paper=True` flag is the actual safety pin against accidentally hitting live markets.**

The `__main__` block prints the paper account summary — useful for quick health checks (`python src/alpaca_connector.py`).

### `src/order_builder.py` — Read MICRO signal, submit order

- `THESES_PATH` = `../tactical_markets/data/theses.jsonl` (cross-project file-on-disk read).
- `NOTIONAL = 10_000` (Phase 1 fixed sizing).
- `latest_signal()` ([src/order_builder.py:16-23](../src/order_builder.py#L16-L23)) iterates the full file and returns the last `signal: true` line. **Does NOT filter by date** — that filtering happens upstream in `run_trading.py:today_signal`.
- `submit_order(thesis)` ([src/order_builder.py:26-43](../src/order_builder.py#L26-L43)) builds `MarketOrderRequest(symbol=thesis["buy"], notional=$10k, side=BUY)` and submits via `trading_client()`.

### `src/trade_logger.py` — Fill polling, ledger writes, NYSE calendar

- `TRADES_PATH` = `data/trades.jsonl`.
- `FILL_POLL_INTERVAL = 2`s, `FILL_POLL_TIMEOUT = 60`s.
- `TERMINAL_FAILED = {REJECTED, CANCELED, EXPIRED}` ([src/trade_logger.py:16](../src/trade_logger.py#L16)) — `wait_for_fill` fails fast on these instead of eating the full timeout.
- `add_trading_days(dt, n)` ([src/trade_logger.py:23-32](../src/trade_logger.py#L23-L32)) — uses `pandas_market_calendars.get_calendar("NYSE").valid_days(...)` to compute the n-th trading day after `dt`. Handles US holidays (Memorial Day, etc.).
- `wait_for_fill(order_id)` ([src/trade_logger.py:35-53](../src/trade_logger.py#L35-L53)) — polls Alpaca every 2s for `OrderStatus.FILLED`. **Does not return on first partial fill** — only on the unambiguous final state. This is the fix from commit `58fa2e1`. Returns `{fill_price, fill_qty, fill_time}`.
- `log_entry(order_result)` ([src/trade_logger.py:56-75](../src/trade_logger.py#L56-L75)) — composes the entry record and appends to `trades.jsonl`. Generates a UUID `trade_id`. Schema documented in [data-models.md](./data-models.md).

### `src/pushover.py` — Notification client

20 LOC. `send(title, message)` → POST to `https://api.pushover.net/1/messages.json` with `PUSHOVER_TOKEN` + `PUSHOVER_USER` from env. If either is missing, prints `[pushover not configured]` and returns `False` — **non-fatal** (mirrors MICRO + MACRO's pattern). Network errors are caught and logged. Returns `True` only on HTTP 200.

### `setup_task.ps1` — Windows Task Scheduler registration

Run once (or after schedule changes) from an elevated PowerShell:

```powershell
PowerShell -ExecutionPolicy Bypass -File .\setup_task.ps1
```

Registers three tasks with `-Force` (safe to re-run):

| Task | Trigger | Action | Settings |
|---|---|---|---|
| `Tactical Trading Wake` | Daily 08:20 AM | `cmd.exe /c exit` | `-WakeToRun -StartWhenAvailable -DontStopIfGoingOnBatteries -AllowStartIfOnBatteries` |
| `Tactical Trading Entry` | Daily 08:35 AM | `.venv\Scripts\python.exe run_trading.py` | Battery flags + `-StartWhenAvailable` |
| `Tactical Trading Exit` | Daily 08:40 AM | `.venv\Scripts\python.exe src\exit_manager.py` | Battery flags + `-StartWhenAvailable` |

Times are CDT — **5 / 10 minutes after CDT market open (08:30 AM CDT)**. Battery flags must be correct from the start (a past session lost a fire on default battery behavior).

## Research subtree (off the production path)

[`research/`](../research/) holds historical backtest scripts. They never place orders, never run on the scheduler, and never read or write `data/trades.jsonl`. Their outputs in `research/data/` are reference material for Phase 2 design decisions — they're cited in [TODO.md](../TODO.md) "Lessons from Phase 1" but not consumed by production code.

- `compare_strategies.py` — 8-strategy head-to-head: buy-and-hold SPY, trend-following SPY (200d MA), 60/40 SPY/TLT, dual momentum (Antonacci GEM), sector momentum top-3 monthly, the live 5d sector rotation, BTC buy-and-hold, BTC + stress overlay. Output: `comparison_summary.csv` + `comparison_report.md`. Window 2014-01-02 to 2026-05-08.
- `sensitivity.py` — 288-combo parameter grid over `position_size × max_positions × hold_days × spread_threshold` for the live sector rotation strategy. Imports from `compare_strategies` to reuse the backtest engine. Output: `sensitivity.csv` + `sensitivity_summary.md`.
- `news_sentiment.py` — prototype that fetches recent ticker-relevant news via Alpaca's `NewsClient`, scores each article with Claude Haiku, aggregates per-ticker daily sentiment. Forward-going only (no historical backfill).

These imports `from compare_strategies import ...` are intra-subtree, not cross-project — so they don't violate the "no cross-sibling imports" rule.

## What's deliberately absent

- **No `tests/` directory.** Phase 1 deferred tests. Inline `if __name__ == "__main__"` smoke runs in each module are the test substitute. The 2026-05-08 hardening pass cleaned up the stale `__main__` block in `trade_logger.py`.
- **No CI/CD config in this project.** The cross-sibling repo at `_genai_tmp/` may have CI; this project is invoked by Windows Task Scheduler, not pipelines.
- **No database.** Append-only JSONL only.
- **No API layer.** The bot does not publish endpoints; it only consumes.
- **No dashboard, no HTML output.** Pushover + stdout only.
- **No `config/` directory yet.** Constants (`NOTIONAL`, `FILL_POLL_*`, `HOLD_DAYS = 5`) live inline in modules. Phase 2 may externalize them.
